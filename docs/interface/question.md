1. 如何做技术选型，考虑哪些因素？

- 一是历史：团队现有的技术栈和沉淀的组件库、函数等，能不能最大程度的利用，减少重复开发；
- 二是未来：选择的技术是否成熟，社区有没有大量的解决方案，能不能满足未来产品的需求；
- 三是成本：团队的熟悉程度有多高，学习成本有多高，会不会影响开发进度？

1. Node.js 流的读写，生产环境会遇到哪些问题？

- 文件锁问题：多个 Node.js 进程/线程同时尝试读取同一个文件。
- 写入冲突：多个 Node.js 进程/线程尝试写入同一个文件，会造成冲突。
- 缓冲区溢出：写入速度过快而缓冲区容量有限，可能会导致缓冲区溢出。

解决方案：

- 实施并发控制，避免文件锁定和写入冲突等问题
- 添加错误监控以及处理（监听 error 事件）

1. 做的技术很广，为什么不深挖一个方向？

- 岗位需求：作为前端负责人，很多场景纯前端是不够的，比如监控、部署、报警、SSR，需要有整体把控的能力。
- 视野有限：面对一个综合问题，纯前端无法给出全面的解决方案，比如性能优化，大部分在网络和后端。

1. 写一道编码题：实现一个带并发限制的 Promise 调度器。

要求同时最多运行的 Promise 数量不超过限制。调度器需要提供两个主要方法：

- `add(promiseCreator)`：添加一个返回 Promise 的函数到调度器。
- `start()`：开始执行所有已添加的 Promise 函数。

4. 怎么去让组员接受你的技术选型，如果推不下去怎么办？

- 首先，我选择一个技术选型，一定是解决项目中的某些问题，或提高开发效率。
- 其次，在这个前提下，我会和组员开会聊技术选型，了解他们的`上手成本`和接受程度。
- 再者，我会在项目中某个不重要的模块先尝试，试一下实际应用，遇到问题继续解决。
- 最后，如果跑通，我们就大规模使用；如果遇到具体难解决的问题，再做评估。
